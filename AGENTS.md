# 仓库规范

## 规则优先级与维护边界
- 本文件用于维护“必须执行”的硬规则与验收门槛。
- `README.md` 仅保留快速入口、命令示例与最小说明，不再维护长篇硬规则。
- 若 `README.md` 与本文件存在冲突，以本文件为准。

## 项目结构与模块组织
本仓库以模板与数据驱动为主（不是传统应用代码仓库）。

- 当前默认运行环境：Windows 11。
- 当前默认终端：PowerShell 7（`pwsh` / PS7）。

- `README.md`：主流程与报告约束说明。
- `template.docx`：最终交付所需的 Word 模板。
- `data/*.xlsx`：各医学主题第4章市场分析的数据源文件；单主题运行时默认读取 `data/<医学主题>.xlsx`。
- `autofile/<医学主题>/`：生成产物目录（不存在时需自动创建）。
- `autofile/<医学主题>/figures/`：全部导出的图表文件（如 `fig_1_1.png`、`fig_4_1.png`）。
- `autofile/<医学主题>/ch04_codex_extract.json`：由 Codex 会话生成的第四章标准化结构化数据。
- `autofile/<医学主题>/backup/`：覆盖写入前的时间戳备份目录。

所有生成文件必须写入 `autofile/` 内，禁止写到仓库根目录。

## 构建、测试与开发命令
当前未配置构建系统。请使用轻量 PowerShell 校验命令：

```powershell
# 创建输出根目录
New-Item -ItemType Directory -Force autofile

# 查看某主题的预期输出
Get-ChildItem "autofile/<医学主题>" -Recurse

# 快速检查关键文件是否存在
@("00_evidence.txt","manifest_text.csv","manifest_fig.csv","ch04_agg_tables.xlsx") |
  ForEach-Object { Test-Path "autofile/<医学主题>/$_" }
```

## 编码风格与命名约定
- 文本文件统一使用 UTF-8 编码，并保留中文医学术语。
- 文件命名需简洁且可复现：
  - 章节文本：`ch01.txt` ... `ch07.txt`
  - 图表文件：`fig_<chapter>_<index>.png`
  - 最终交付：`《<医学主题>市场分析报告》_final.docx`
- 各类清单文件（manifest）需保持机器可读（`.csv`、稳定字段名、避免临时重命名）。

## 测试与验证规范
当前没有自动化测试框架。请将验证视为发布闸门：

- 确认 `autofile/<医学主题>/` 下必需文件齐全。
- 确认第4章图表均来源于 Excel 数据，并且标签一致。
- 确认覆盖写入会在 `backup/` 下生成备份。
- 打开最终 `.docx`，检查目录、标题层级、图注与数据来源行是否正确渲染。

## 正式交付字数与结构闸门（强制）
- 正式交付版报告总字数必须控制在 `30000-34000`（按章节正文 + `summary.txt` 合并计算，去空白后统计）。
- 各章最低字数门槛如下：
  - 第1章：`3000`
  - 第2章：`3500`
  - 第3章：`4800`
  - 第4章：`3000`
  - 第5章：`4800`
  - 第6章：`4800`
  - 第7章：`4800`
- 当前发布闸门允许“低于章节下限 `100` 字内通过”；超过 `100` 字视为未达标，必须补写。
- 各章正文必须由当前 Codex 会话直接写入 `ch01.txt` ... `ch07.txt`，不得由脚本补写、拼接或兜底生成。
- 各非小结正文 block 必须显式写出三级标题，供模板中的 `三级目录` 使用；不得只停留在二级标题。
- 各章“本章小结”应控制在约 `500` 字，建议区间 `450-650` 字。
- “本章小结”不得插入图表，不得写成偏企业管理建议的长篇段落。
- 第4章必须严格基于 Excel 与 `ch04_codex_extract.json` 写作；若某个渠道级明细数据在源 Excel 中缺失，则应省略对应小节与图表，不得用占位分析补齐。
- 第4章图表若关键指标无有效值，也应直接省略，不得输出空图、空柱或仅保留标题占位。
- 上述“无有效值”包括但不限于：对应渠道最新期销售额全为 `0`、同比/增速分母为 `0` 导致不可计算、Top10 最新期全为 `0`、CR5 无法形成有效值。
- 第4章排名/格局类图表（如 Top10、活跃品牌、活跃企业）若有效正值主体少于 `2` 个，也应直接省略；单一主体自成一图不视为有效竞争格局展示。
- 第4章 `CR5` 或其他集中度图表若仅因单一有效主体导致机械性等于 `100%`，也不视为有效竞争格局，应直接省略，不得把“单主体垄断式空格局”误写成可分析的竞争态势。
- 当某张第4章图表被省略时，正文对应小节与相关表述也必须按数据可用性同步裁剪，不得保留“理论上应有但当前无数据”的占位分析。

## 提交与合并请求规范
当前目录没有可用 Git 历史，后续请按以下标准执行：

- 提交信息格式：`type(scope): summary`（示例：`docs(readme): clarify chapter-4 outputs`）。
- 每次提交应聚焦单一变更主题（模板改动、数据改动、示例产物尽量分开）。
- PR 需包含：变更目的、修改文件列表、已执行验证步骤、版式改动前后截图。
- 有任务/问题编号时请在 PR 中关联。

## Codex 优先 AI 策略（强制）
- 本仓库中的“AI主导”定义为：**由当前 Codex 会话模型主导写作与语义决策**。
- Python 脚本仅做辅助：数据整理、图表渲染、docx 装配、QA 校验、备份与清单处理。
- 禁止新增或恢复任何需要 API 凭证的脚本侧外部模型调用。
- 正常工作流中不得要求设置 `OPENAI_API_KEY`。
- 禁止恢复 `--gen-mode ai` 或其他脚本内 LLM 生成路径。
- 默认协作方式：用户仅提供医学主题与第4章 Excel 数据；Codex 负责写作，并负责将第4章 Excel 提取为 `ch04_codex_extract.json`；脚本负责装配与校验。
- 当用户要求“按 AGENTS.md 一键全自动生成正式交付版报告”时，默认含义是：**必须以正式交付质量为目标执行完整流程**，而不是先生成测试稿、保底稿或占位稿。
- “批量一键生成”默认采用**最稳妥路径**：逐主题完成正式正文、`fig23_codex_spec.json`、图表、docx 与 QA；只有单个主题 `qa_check.txt` 明确通过，才算该主题完成。
- 严禁为了追求批量速度，擅自使用临时拼接式正文生成器、一次性草稿脚本、保底模板稿或其他未进入仓库正式流程的临时方案替代 Codex 写作。
- 若批量流程与正式交付质量发生冲突，必须优先保证质量，不得以“先跑通再说”为理由降低正文质量标准。
- 禁止在仓库根目录留下 `_tmp_*.py`、临时批量脚本或一次性调试产物；所有可复用流程必须进入正式代码或说明文件。

## 图表 QA 工作流（强制）
- 图表生成后，必须先执行脚本侧 QA（确定性规则，见 `qa_check.txt`）。
- 在最终交付前，必须由 **Codex 会话** 对关键机制图做视觉复核（重点是 `fig_2_3.png`）。
- 图表图片应统一按 Word 插入宽度导出，避免因原始宽度不同导致插入到 `.docx` 后缩放比例不一致。
- 图表图片中的 `图表x-x：xxx` 主标题与 `数据来源：xxxx` 来源行必须使用统一字体与字号；图内数据层文字可更小，但必须保持可读。
- 流程图、时间线图、关系图不得强行单排；当节点或事件标签过长时，必须优先换行、多排或改布局，禁止出现文本重叠。
- 不得对所有图表硬套同一布局模板；至少应区分定量图、流程图、时间线图与关系/机制图。
- 脚本与 Codex 职责拆分如下：
  - 脚本：结构化规则检查（标题/来源一致性、语义约束、几何与可读性规则等）。
  - Codex：图像级临床逻辑复核，并给出最终通过/驳回判断。
- 注意：Python 脚本无法直接调用 Codex；视觉复核必须在 Codex 会话中显式触发。
